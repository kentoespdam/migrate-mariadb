## Ringkasan Eksekutif

| Severity | Jumlah temuan | Kategori dominan |
|---|---|---|
| 🔴 Critical | 4 | Data correctness, cursor reuse, resource leak |
| 🟠 High | 6 | Thread-safety, retry logic, transaction lifecycle |
| 🟡 Medium | 6 | Caching, dry-run reporting, async/sync mixing |
| ⚪ Low | 4 | Style/Ruff, dead code, placeholder feature |

**Risiko terbesar yang harus diperbaiki segera:**
1. `MigrationResult.failed_batches` tidak pernah di-increment → migrasi yang sebagian gagal dilaporkan sebagai "success".
2. `SSCursor` di-share lintas tabel → potensi `Unread result found` / data corruption pada migrasi multi-tabel.
3. Cancellation tidak menutup koneksi/transaksi target → resource leak di server.
4. Worker thread Textual memutasi widget langsung → race & potensi crash UI.

## A. `pysync_maria/db/engine.py`

### 🔴 A1 — `failed_batches` tidak di-increment, status selalu "success"
**Symptom**
- Walau `batch_err` diisi pada line 148 atau 161, evaluasi status pada line 181 (`if res.failed_batches == 0: res.status = "success"`) selalu benar.
- Caller (TUI) menampilkan ✅ pada tabel yang sebenarnya gagal sebagian.

**Root Cause**
- Setelah blok retry selesai, `batch_err` hanya disisipkan ke `BatchResult.error` lalu dikirim ke callback. Tidak ada `res.failed_batches += 1` maupun `res.errors.append(...)`.

**Proposed Fix (Ruff/PEP8)**
```python
# engine.py — setelah line 174 (sebelum membangun batch_res)
if batch_err:
    res.failed_batches += 1
    res.errors.append(f"Batch {batch_num}: {batch_err}")
```

---

### 🔴 A2 — SSCursor di-share antar tabel
**Symptom**
- `migration_screen.py:80` membuka *satu* `src_cursor` (SSCursor) lalu meneruskannya berulang ke `migrate_table()` per tabel.
- Setelah `stream_table()` mengeksekusi `SELECT` baru pada cursor yang masih punya state dari tabel sebelumnya, MariaDB melempar `Commands out of sync` / `Unread result found` atau koneksi diputus.

**Root Cause**
- `stream_table(cursor, table, ...)` memanggil `cursor.execute(query)` tanpa close/recreate. SSCursor unbuffered tidak otomatis membersihkan result-set lama.

**Proposed Fix**
```python
# engine.py — refactor signature
def migrate_table(
    src_conn: Any,
    tgt_conn: Any,
    table: str,
    ...
) -> MigrationResult:
    src_cursor = src_conn.cursor(cursor_class=SSCursor)
    try:
        for batch_rows in stream_table(src_cursor, table, source_cols, batch_size):
            ...
    finally:
        src_cursor.close()
```
Caller (`migration_screen.py`) cukup memberi `src_conn`; engine yang mengelola cursor lifecycle.

### 🟠 A3 — Bare `except:` pada ping retry (Ruff E722)
**Symptom**
- Line 156-157: `except: pass` menelan semua error reconnect tanpa log.

**Root Cause**
- Coding shortcut. Melanggar Ruff `E722` & PEP8 §"Programming Recommendations".

**Proposed Fix**
```python
try:
    target_conn.ping(reconnect=True)
except mysql.connector.Error as ping_err:
    logger.warning(
        f"Reconnect failed in batch {batch_num}: {ping_err}"
    )
```

### 🟠 A4 — Off-by-one pada loop retry
**Symptom**
- Line 138 menggunakan `while retry_count <= max_retries`. Untuk `max_retries=3`, loop melakukan 4 percobaan (counter 0 → 4) sebelum break di line 149.

**Root Cause**
- Kondisi loop dan increment-then-check tidak konsisten.

**Proposed Fix**
```python
retry_count = 0
last_err: Optional[Exception] = None
while retry_count < max_retries:
    try:
        ...
        break
    except (mysql.connector.OperationalError,
            mysql.connector.InterfaceError) as e:
        last_err = e
        retry_count += 1
        wait_time = 2 ** retry_count
        logger.warning(
            f"Retry {retry_count}/{max_retries} for batch "
            f"{batch_num} after {wait_time}s: {e}"
        )
        time.sleep(wait_time)
        try:
            target_conn.ping(reconnect=True)
        except mysql.connector.Error as ping_err:
            logger.warning(f"Reconnect failed: {ping_err}")
else:
    batch_err = f"Failed after {max_retries} retries: {last_err}"
```

### 🟠 A5 — Critical exception ditelan tanpa rollback
**Symptom**
- Line 188-191: `except Exception` hanya log dan set `status="failed"`. `target_conn` mungkin meninggalkan transaksi terbuka.

**Root Cause**
- Tidak ada cleanup transaksional pada path error luar loop.

**Proposed Fix**
```python
except Exception as e:
    res.status = "failed"
    res.errors.append(f"Critical error: {e!s}")
    logger.exception(f"Critical error migrating {table}")
    try:
        if not dry_run:
            target_conn.rollback()
    except mysql.connector.Error as rb_err:
        logger.error(f"Rollback failed for {table}: {rb_err}")
```

### 🟡 A6 — Column mapping diam-diam membuang kolom
**Symptom**
- Line 120-121: `[column_map[c] for c in columns_a if column_map.get(c)]` membuang kolom unmapped tanpa peringatan → potensi *silent data loss*.

**Root Cause**
- Tidak ada pre-flight validation atas hasil mapping.

**Proposed Fix**

```python
unmapped = [c for c in columns_a if not column_map.get(c)]
if unmapped:

    logger.warning(msg)
    res.warnings.append(msg)  # MigrationResult.warnings (lihat Architecture #5)
```

## B. `pysync_maria/db/connection.py`

### 🟠 B1 — Cursor leak di `get_streaming_connection`
**Symptom**
- Line 73 membuat `cursor`. Line 77 `cursor.close()` *setelah* `yield`. Bila konsumen melempar exception, line 77 terlewati; `finally` (line 80-82) hanya menutup koneksi.

**Root Cause**
- Cleanup cursor tidak berada di `finally`.

**Proposed Fix**
```python
@contextmanager
def get_streaming_connection(config: HostConfig):
    cnx = None
    cursor = None
    try:
        cnx = mysql.connector.connect(...)
        cnx.ping(reconnect=True, attempts=3, delay=1)
        cursor = cnx.cursor(cursor_class=SSCursor)
        yield cnx, cursor
    except mysql.connector.Error as err:
        raise ConnectionError(
            f"MariaDB Streaming Error [{err.errno}]: {err.msg}"
        ) from err
    finally:
        if cursor is not None:
            try:
                cursor.close()
            except mysql.connector.Error:
                pass
        if cnx is not None and cnx.is_connected():
            cnx.close()
```

**Symptom**

**Proposed Fix**

## C. `pysync_maria/db/metadata.py`

### 🔴 C1 — `@lru_cache` pada parameter koneksi (line 52, 79, 103)
**Symptom**
- Decorator caching memakai `conn` (objek `MySQLConnection`) sebagai key. Implikasi:
  1. Cache dihash via `id()` → koneksi baru tak pernah hit (pemborosan).
  2. Bila objek koneksi yang sama di-`reconnect()` ke database/schema lain, cache mengembalikan **metadata stale** dari schema lama.
  3. Cache memegang strong reference ke koneksi yang seharusnya sudah ditutup → resource leak.

**Root Cause**
- `lru_cache` tidak cocok untuk parameter dengan lifecycle eksternal (mutable resource).

**Proposed Fix**
- **Hapus seluruh `@lru_cache`** dari `get_tables`, `get_columns`, `get_foreign_keys`. Query information_schema cepat; caching tidak berguna dan berbahaya.
- Bila ke depan dibutuhkan memo, lakukan di lapisan TUI dengan key `(database, table)` dan invalidasi eksplisit saat koneksi di-reset.

### 🟡 C2 — Cursor type campuran (dictionary vs tuple)
**Symptom**
- Line 67 & 118 pakai `cursor(dictionary=True)`; line 88 pakai tuple-cursor.

**Proposed Fix**
- Standarkan ke `dictionary=True` di seluruh metadata helpers untuk konsistensi & readability.

## D. `pysync_maria/tui/screens/migration_screen.py`

### 🔴 D1 — Cancellation tidak menghentikan engine, koneksi tidak ditutup
**Symptom**
- `cancel-btn` hanya memanggil `self.workers.cancel_all()` (line 163). Engine `migrate_table` blocking di `executemany`/`fetchmany` tidak punya hook cancellation. Saat thread akhirnya dibebaskan, transaksi target dapat tergantung.

**Root Cause**
- Tidak ada cooperative cancellation flag yang dibaca engine.

**Proposed Fix**
```python
# migration_screen.py
import threading
self.cancel_event = threading.Event()
self.pause_event = threading.Event(); self.pause_event.set()  # set=running

# Dalam on_button_pressed cancel-btn:
self.cancel_event.set()

# Dalam run_migration:
res = migrate_table(
    src_conn=src_conn,
    tgt_conn=tgt_conn,
    ...,
    cancel_event=self.cancel_event,
    pause_event=self.pause_event,
)

# engine.py: setiap awal batch
if cancel_event is not None and cancel_event.is_set():
    res.cancelled = True
    if not dry_run:
        target_conn.rollback()
    break
if pause_event is not None:
    pause_event.wait()  # block while paused
```

Tambahkan `try/finally` di worker untuk memastikan `tgt_conn.rollback()` dipanggil pada path cancel.

## E. `pysync_maria/tui/screens/connection_screen.py` & `table_select_screen.py`

### 🔴 E1 — Worker thread memutasi widget secara langsung
**Symptom**
- `table_select_screen.py:49,82` — `table_list.loading = True/False` di thread worker.
- `connection_screen.py:59` — `form.query_one("#status", Label)` dipanggil di thread.

**Root Cause**
- Pelanggaran kontrak Textual: widget mutation harus di main thread.

**Proposed Fix**
```python
# table_select_screen.py
self.app.call_from_thread(setattr, table_list, "loading", True)
...
self.app.call_from_thread(setattr, table_list, "loading", False)

# connection_screen.py — pindahkan query widget ke main-thread callback
def _set_status(text: str, color: str) -> None:
    self.query_one("#status", Label).update(f"[{color}]{text}[/]")
self.app.call_from_thread(_set_status, "Connected", "green")
```

### 🟠 E2 — Race pada `self.app.source_config`/`target_config`
**Symptom**
- Worker tulis config dari `connection_screen.py:76-79`. Main thread baca pada `check_all_ready`. Tidak ada lock.

**Proposed Fix**
- Tulis config hanya melalui `call_from_thread` ke method handler di main thread:
```python
def _commit_config(role: str, cfg: HostConfig) -> None:
    setattr(self.app, f"{role}_config", cfg)
    self.check_all_ready()
self.app.call_from_thread(_commit_config, "source", config)
```

### 🟡 E3 — Sync DB call di async context
**Symptom**
- `table_select_screen.py:180-186` — `get_columns(...)` sinkron di dalam `async def fetch_and_open_mapping`.

**Proposed Fix**
```python
import asyncio
cols = await asyncio.to_thread(
    get_columns, conn, self.app.source_config.database, table_name
)
```
atau bungkus dalam `@work(thread=True)` dan post message hasilnya.

## Architecture Decisions

> Bagian ini berisi keputusan refactor terstruktur yang melampaui patch lokal.

### AD-1. Pindahkan kepemilikan SSCursor ke engine
Tanda tangan `migrate_table` berubah dari `(conn_a: Tuple[conn, cursor], conn_b)` menjadi `(src_conn, tgt_conn)`. Engine yang membuat & menutup SSCursor per tabel. Menyelesaikan **A2** secara struktural.

### AD-2. Cooperative cancellation & pause via `threading.Event`
Tambah parameter opsional `cancel_event` & `pause_event` ke `migrate_table`. Engine memeriksanya di awal setiap iterasi batch. TUI kontrol via tombol. Memungkinkan implementasi D1 & D4 dengan benar.

### AD-3. Ekstrak retry helper
Buat `pysync_maria/db/_retry.py`:
```python
def retry_with_backoff(
    fn: Callable[[], T],
    *,
    retries: int = 3,
    on_exceptions: Tuple[type[Exception], ...] = (
        mysql.connector.OperationalError,
        mysql.connector.InterfaceError,
    ),
    on_retry: Optional[Callable[[int, Exception], None]] = None,
) -> T: ...
```
`migrate_table` jadi linear & lebih mudah ditest.

### AD-4. Hapus `@lru_cache` dari metadata
Caching untuk objek dengan lifecycle eksternal harus eksplisit di consumer, bukan via decorator.

### AD-5. Perluas `MigrationResult`
```python
@dataclass
class MigrationResult:
    table_name: str
    total_rows_read: int = 0
    total_rows_written: int = 0
    total_batches: int = 0
    failed_batches: int = 0
    elapsed_seconds: float = 0.0
    status: str = "success"
    errors: List[str] = field(default_factory=list)
    warnings: List[str] = field(default_factory=list)
    dry_run: bool = False
    cancelled: bool = False
```
Gunakan `field(default_factory=list)` ketimbang pola `__post_init__` untuk konsistensi PEP8.

### AD-6. Logger setup terpusat
Konfigurasi root logger di `main.py` dengan `RotatingFileHandler` ke `logs/pysync_maria.log` (sudah ada folder `logs/`). Hindari pengaturan logger ad-hoc per modul.

### AD-7. Tambah konfigurasi Ruff di `pyproject.toml`
```toml
[tool.ruff]
target-version = "py312"
line-length = 100

[tool.ruff.lint]
select = ["E", "F", "W", "I", "B", "UP", "SIM", "RUF"]
ignore = []
```
Jalankan `uv run ruff check --fix pysync_maria tests` & `uv run ruff format pysync_maria tests` setelah patch.

## Verification Plan

1. **Static analysis**
   - `uv run ruff check pysync_maria tests`
   - `uv run ruff format --check pysync_maria tests`

2. **Unit tests**
   - `uv run pytest tests/ -v`
   - Tambahan: failure-injection test, multi-table SSCursor test, cancellation test (lihat §F).

3. **Manual smoke (TUI)**
   - `uv run python -m pysync_maria.main` dengan `.env.source` & `.env.target` ke dua MariaDB lokal.
   - Skenario: 1 tabel kecil (~100 rows) + 1 tabel besar (≥100k rows).
   - Verifikasi:
     - Progress bar update halus, `speed > 0`.
     - Cancel di tengah migrasi → engine berhenti dalam ≤ 1 batch; `SHOW PROCESSLIST` di target tidak meninggalkan thread idle dengan transaksi terbuka.
     - Dry-run: `SELECT COUNT(*)` target sebelum/sesudah identik.

4. **Memory verification**
   - Migrasi tabel ≥1M rows; pantau RSS proses dengan `psutil` — wajib stabil < 200 MB selama migrasi (membuktikan SSCursor streaming + tidak ada akumulasi list).

5. **Concurrency stress**
   - Jalankan migrasi, sembari simulasi pemutusan jaringan ke target (drop koneksi via `iptables`/Docker net pause). Verifikasi retry log muncul, retry berhasil, dan `failed_batches` bertambah saat retry habis.

## Out of Scope

- Migrasi DDL (CREATE TABLE) — diasumsikan target schema sudah ada.
- Tabel partitioned / sharded.
- Tuning network (compress, SSL).
- Migrasi triggers, stored procedures, views, events.
